title: WebSocket transport
---

Apollo iOS supports executing GraphQL operations over [WebSocket](https://developer.mozilla.org/en-US/docs/Web/API/WebSocket) using the [`graphql-transport-ws`](https://github.com/enisdenjo/graphql-ws/blob/master/PROTOCOL.md) protocol. This is the recommended approach for GraphQL subscriptions and is provided by the `ApolloWebSocket` library.

## Installation

`WebSocketTransport` is included in the `ApolloWebSocket` library, which is part of the Apollo iOS SDK but is not included by default.

Add the `ApolloWebSocket` library to your project via Swift Package Manager:

1. In Xcode, select **File > Add Package Dependencies…**
2. Enter the Apollo iOS package URL and select the `ApolloWebSocket` library target.
3. Import `ApolloWebSocket` where needed:

```swift
import ApolloWebSocket
```

## Creating a WebSocket client

### Subscriptions over WebSocket, queries and mutations over HTTP

The most common configuration routes subscription operations through a WebSocket connection while sending queries and mutations over HTTP. Use `SplitNetworkTransport` to combine a `RequestChainNetworkTransport` (HTTP) with a `WebSocketTransport` (WebSocket):

```swift
import Apollo
import ApolloWebSocket

let store = ApolloStore()

let httpTransport = RequestChainNetworkTransport(
  interceptorProvider: DefaultInterceptorProvider(store: store),
  endpointURL: URL(string: "https://example.com/graphql")!
)

let wsTransport = try WebSocketTransport(
  urlSession: URLSession.shared,
  store: store,
  endpointURL: URL(string: "wss://example.com/graphql")!
)

let splitTransport = SplitNetworkTransport(
  queryTransport: httpTransport,
  mutationTransport: httpTransport,
  subscriptionTransport: wsTransport
)

let client = ApolloClient(networkTransport: splitTransport, store: store)
```

> **Important:** Pass the same `ApolloStore` instance to both the `WebSocketTransport` and `ApolloClient`. The transport uses the store to read cached data before network fetches and to write server responses back to the cache.

### All operations over WebSocket

You can also send queries, mutations, and subscriptions all through a single WebSocket connection. Use `WebSocketTransport` directly as the `NetworkTransport`:

```swift
let store = ApolloStore()

let wsTransport = try WebSocketTransport(
  urlSession: URLSession.shared,
  store: store,
  endpointURL: URL(string: "wss://example.com/graphql")!
)

let client = ApolloClient(networkTransport: wsTransport, store: store)
```

> **Note:** Queries and mutations executed over WebSocket are terminated immediately if the connection drops, since replaying a mutation could cause duplicate side effects. Only subscriptions survive reconnection.

## Configuration

`WebSocketTransport` accepts a `Configuration` struct that controls its behavior:

```swift
let config = WebSocketTransport.Configuration(
  reconnectionInterval: 5,    // seconds; -1 to disable
  connectingPayload: ["Authorization": "Bearer \(token)"],
  pingInterval: 20            // seconds; nil to disable
)

let wsTransport = try WebSocketTransport(
  urlSession: URLSession.shared,
  store: store,
  endpointURL: URL(string: "wss://example.com/graphql")!,
  configuration: config
)
```

### Configuration options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `reconnectionInterval` | `TimeInterval` | `-1` (disabled) | Seconds to wait before reconnecting after a disconnect. `0` reconnects immediately. Negative values disable auto-reconnection. |
| `connectingPayload` | `JSONEncodableDictionary?` | `nil` | Payload sent in the `connection_init` message. Commonly used for authentication. |
| `pingInterval` | `TimeInterval?` | `nil` (disabled) | Interval at which the client sends `ping` keepalive messages to the server. |
| `requestBodyCreator` | `JSONRequestBodyCreator` | `DefaultRequestBodyCreator()` | Builds the JSON payload for `subscribe` messages. |
| `operationMessageIdCreator` | `OperationMessageIdCreator` | `ApolloSequencedOperationMessageIdCreator()` | Generates unique IDs for each operation message. |
| `clientAwarenessMetadata` | `ClientAwarenessMetadata` | `ClientAwarenessMetadata()` | Adds Apollo client name/version headers to the WebSocket handshake request. |

## Authentication

### Connection-level authentication

For GraphQL servers that accept auth tokens in the `connection_init` payload (the most common WebSocket auth pattern):

```swift
let config = WebSocketTransport.Configuration(
  connectingPayload: ["Authorization": "Bearer \(userToken)"]
)
```

### Header-based authentication

Some servers authenticate the WebSocket upgrade request via HTTP headers. Pass a custom `URLSession` with the required headers, or update headers at runtime:

```swift
await wsTransport.updateHeaderValues(
  ["Authorization": "Bearer \(newToken)"],
  reconnectIfConnected: true  // force reconnect to apply new headers immediately
)
```

### Updating auth at runtime

When a user's auth token changes (e.g., after a token refresh), update the transport without losing active subscriptions:

```swift
// Update the connection_init payload
await wsTransport.updateConnectingPayload(
  ["Authorization": "Bearer \(newToken)"],
  reconnectIfConnected: true
)
```

Setting `reconnectIfConnected: true` disconnects the current connection and reconnects immediately with the new payload. Active subscriptions survive the reconnection when auto-reconnection is enabled.

## Auto-reconnection

When `reconnectionInterval` is set to a non-negative value, the transport automatically reconnects and re-subscribes active subscriptions when the connection drops.

```swift
let config = WebSocketTransport.Configuration(
  reconnectionInterval: 3  // wait 3 seconds before reconnecting
)
```

Key behaviors during auto-reconnection:
- **Subscription streams remain open.** Callers iterating `for try await result in subscription` are unaware the disconnect happened — new results resume flowing after reconnection.
- **Queries and mutations are terminated immediately.** They are not retried across a reconnection.
- **If reconnection fails**, all remaining subscriber streams are finished with an error.

## Pause and resume

Use `pause()` and `resume()` to gracefully suspend and restore the WebSocket connection — for example, when your app enters the background:

```swift
class AppLifecycleObserver {
  let transport: WebSocketTransport

  func applicationDidEnterBackground() {
    Task { await transport.pause() }
  }

  func applicationWillEnterForeground() {
    Task { await transport.resume() }
  }
}
```

During a pause:
- The underlying WebSocket connection is closed.
- **Subscription streams remain alive.** Subscriptions are automatically re-subscribed when `resume()` is called and the connection is re-established.
- **Queries and mutations that are in-flight are terminated.** They cannot safely survive a connection interruption.
- Auto-reconnection is suppressed — the transport waits for an explicit `resume()` call.

## Subscription lifecycle state

WebSocket subscriptions expose a `state` property via `SubscriptionStream` that reflects the subscription's current lifecycle position:

```swift
let subscription = try client.subscribe(subscription: ReviewAddedSubscription())

print(subscription.state) // .pending

Task {
  for try await result in subscription {
    print(subscription.state) // .active
    // handle result
  }
  print(subscription.state) // .finished(.completed)
}
```

### States

| State | Description |
|-------|-------------|
| `.pending` | Initiated but not yet active. The transport may be connecting or sending the subscribe message. |
| `.active` | Active. The subscription can receive data from the server. |
| `.reconnecting` | The connection was lost. The transport is attempting to reconnect. |
| `.paused` | The connection was intentionally paused via `transport.pause()`. |
| `.finished(.completed)` | Ended normally (server sent `complete`). |
| `.finished(.cancelled)` | Cancelled by the client. |
| `.finished(.error(error))` | Terminated due to an error. |

The `.reconnecting` and `.paused` states are specific to `WebSocketTransport`. They are never set by other transport implementations.

## Delegate

Implement `WebSocketTransportDelegate` to observe connection lifecycle events:

```swift
class MyTransportDelegate: WebSocketTransportDelegate {
  func webSocketTransportDidConnect(_ transport: isolated WebSocketTransport) {
    print("WebSocket connected")
  }

  func webSocketTransportDidReconnect(_ transport: isolated WebSocketTransport) {
    print("WebSocket reconnected — all active subscriptions have been re-subscribed")
  }

  func webSocketTransport(
    _ transport: isolated WebSocketTransport,
    didDisconnectWithError error: (any Error)?
  ) {
    if let error {
      print("WebSocket disconnected with error: \(error)")
    } else {
      print("WebSocket disconnected cleanly")
    }
  }
}

// Attach the delegate
await wsTransport.setDelegate(myDelegate)
```

> **Note:** Delegate methods receive the `WebSocketTransport` as an `isolated` parameter, meaning they run within the transport's actor isolation domain. If your delegate performs work that should not block the transport's receive loop, dispatch a `Task` inside the implementation:
> ```swift
> func webSocketTransportDidConnect(_ transport: isolated WebSocketTransport) {
>   Task { await self.updateUI() }
> }
> ```

### Delegate methods

| Method | Description |
|--------|-------------|
| `webSocketTransportDidConnect(_:)` | Called after the initial `connection_ack` from the server. |
| `webSocketTransportDidReconnect(_:)` | Called after a subsequent `connection_ack` following a reconnection. |
| `webSocketTransport(_:didDisconnectWithError:)` | Called when the connection drops. `error` is `nil` for a clean disconnect. |
| `webSocketTransport(_:didReceivePingWithPayload:)` | Called when the server sends a `ping`. The transport automatically replies with `pong`. |
| `webSocketTransport(_:didReceivePongWithPayload:)` | Called when the server sends a `pong` in response to a client ping. |

## Eager connection

By default, `WebSocketTransport` opens the WebSocket connection lazily on the first `subscribe` call. To open the connection eagerly before any operations:

```swift
let wsTransport = try WebSocketTransport(
  urlSession: URLSession.shared,
  store: store,
  endpointURL: URL(string: "wss://example.com/graphql")!
)

// Eagerly start the connection handshake
await wsTransport.resume()

// Subsequent subscribe() calls reuse the already-established connection
let client = ApolloClient(networkTransport: wsTransport, store: store)
```

## Keepalive pings

Some WebSocket servers drop idle connections if they don't receive data within a certain window. Enable client-initiated pings to keep the connection alive:

```swift
let config = WebSocketTransport.Configuration(
  pingInterval: 20  // send a ping every 20 seconds
)
```

Pings are only sent after the server sends `connection_ack`. The timer pauses on disconnect or `pause()` and restarts on reconnect or `resume()`.

## Custom operation message IDs

By default, operations are identified by sequential integers (`"1"`, `"2"`, `"3"`, ...). To use custom identifiers, implement `OperationMessageIdCreator`:

```swift
struct UUIDMessageIdCreator: OperationMessageIdCreator {
  mutating func requestId() -> String {
    UUID().uuidString
  }
}

let config = WebSocketTransport.Configuration(
  operationMessageIdCreator: UUIDMessageIdCreator()
)
```

## Testing

`WebSocketTransport` accepts any type conforming to `WebSocketURLSession`, making it easy to inject mock WebSocket tasks in tests:

```swift
class MockWebSocketURLSession: WebSocketURLSession {
  func webSocketTask(with request: URLRequest) -> any WebSocketTask {
    MockWebSocketTask()
  }
}
```

Pass your mock session when creating the transport:

```swift
let transport = try WebSocketTransport(
  urlSession: MockWebSocketURLSession(),
  store: ApolloStore(),
  endpointURL: URL(string: "wss://example.com/graphql")!
)
```

## Error handling

`WebSocketTransport` surfaces errors through the `AsyncThrowingStream` returned by operations. Errors from `WebSocketTransport.Error` indicate transport-level failures:

```swift
do {
  let subscription = try client.subscribe(subscription: ReviewAddedSubscription())
  for try await result in subscription {
    // handle result
  }
} catch WebSocketTransport.Error.connectionClosed {
  // The connection closed before the server acknowledged it.
} catch WebSocketTransport.Error.graphQLErrors(let errors) {
  // The server returned GraphQL errors for the operation.
} catch WebSocketTransport.Error.unrecognizedMessage {
  // A message was received that doesn't conform to graphql-transport-ws.
} catch {
  // Other errors (e.g., network connectivity).
}
```
